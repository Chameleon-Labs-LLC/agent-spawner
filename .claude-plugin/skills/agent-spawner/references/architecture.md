# Architecture reference

The layered agent stack that the scaffolder implements. The shape is borrowed from multi-agent "mission control" designs — adapted for a public, Linux-first setup using the Anthropic Managed Agents beta plus the Agent SDK.

## The six layers

| Layer | What it is | Scaffolder's role |
|---|---|---|
| 1. UI | Phone, browser, dashboard | Not generated — the scaffolder produces agents and channels that UIs talk to. |
| 2. Channels | Telegram, Slack, Discord, Dashboard | One adapter per requested channel, each implementing the four-ring security model (see `channels.md`). |
| 3. Core engine | Agent runtime + tools + message pipeline | A single 8-stage pipeline inside `local/agent.py`. Managed counterparts run the same shape in the cloud. |
| 4. Subsystems | Memory, audit log, hive mind | Audit log generated per-agent. Hive mind is an optional shared SQLite file. |
| 5. Security | Allowlist → PIN → Idle Lock → Kill Phrase → Exfil Guard → Audit Log, plus HMAC for bridge calls | All generated by default; see `channels.md` and `managed-vs-local.md`. |
| 6. Infra | systemd-user (primary), Windows Task Scheduler (secondary), Python 3.11+ with `uv` | Both autostart unit files are generated. |

## The 8-stage message pipeline

```
incoming message → channel adapter → queue → classifier → memory inject
                → Agent SDK / Managed Agents session → exfil guard → cost footer → reply
```

Every generated agent implements all eight stages — even local ones. This keeps the message lifecycle uniform so a local and a managed agent are interchangeable from the channel's point of view.

## The persona spec card

Every agent has a `persona.json` with:

- `name` — snake_case, used everywhere as the agent ID
- `voice_id` / `gemini_voice` — reserved for a future voice layer; kept so personas don't need editing later
- `system_prompt` — the role definition
- `tools` — `delegate_to_agent`, `answer_as_agent`, `get_time`, `list_agents`, plus custom tools
- `mcp_servers` — list of `{type: "url", url, name}` entries
- `channels` — subset of `telegram`, `slack`, `discord`

## Hive mind (optional)

If enabled, each agent appends a row per completed task to a shared SQLite database (default: `~/.agents/hive.db`):

```
agent_id, task_id, started_at, completed_at, summary, cost_usd, tokens_in, tokens_out
```

For managed agents, the hive mind is updated via a webhook the managed agent posts back to a local sync daemon (separate infra — out of scope for the per-agent scaffolder, but the agents are wired to call it).

## Why both Managed Agents and the Agent SDK?

- **Managed Agents** — deployment/orchestration for customer-facing or always-on surfaces. Runs in Anthropic's cloud.
- **Agent SDK** — the same agent shape running locally for development, testing, private workflows, or work that needs access to resources that aren't publicly reachable.
- **Hybrid** — a local agent that delegates high-trust or context-rich tasks to its managed twin over an HMAC-signed HTTPS call.

See `managed-vs-local.md` for the bridge protocol.
